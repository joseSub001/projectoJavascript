# Proyecto Final del Curso: JavaScript Avanzado

El presente proyecto tiene como finalidad que los estudiantes integren los conocimientos adquiridos durante el curso mediante el desarrollo de una aplicación web innovadora que resuelva una problemática real utilizando tecnologías modernas del ecosistema JavaScript.

---

## 1. Objetivo General

Diseñar, desarrollar y desplegar una aplicación web funcional que implemente un flujo completo de negocio aplicando buenas prácticas de desarrollo, metodologías ágiles y tecnologías basadas en JavaScript.

---

## 2. Tecnologías Permitidas

Los equipos podrán seleccionar una de las siguientes alternativas:

* **Angular**
* **React**
* **JavaScript puro (Vanilla JavaScript)**

*Nota: La elección tecnológica deberá justificarse durante la presentación del Sprint 0.*

---

## 3. Requisitos Mínimos del Proyecto

* Propuesta innovadora orientada a resolver una necesidad real.
* Flujo completo de negocio.
* Gestión de usuarios o actores.
* Operaciones CRUD.
* Consumo de APIs.
* Persistencia de información.
* Validaciones de entrada.
* Dashboard o indicadores.
* Uso de Git y GitHub.
* Aplicación desplegada en un entorno accesible.

---

## 4. Organización del Trabajo

Cada equipo trabajará utilizando un enfoque basado en **Scrum**. Durante cada semana deberán presentar evidencia del avance de su iniciativa mediante una demostración funcional del incremento desarrollado.

## 6. Detalle de Entregables por Semana

### Semana 11 – Definición de la Iniciativa

* **Objetivo:** Identificar una oportunidad de negocio o problemática real que será resuelta mediante una aplicación web.
* **Entregables:**
1. **Formación de Equipos:**
* Equipos de 2 a 4 integrantes.
* Definición de roles preliminares.


2. **Definición de la Iniciativa (Debe responder):**
* ¿Qué problema resuelve?
* ¿Por qué es importante?
* ¿Qué valor genera?


3. **Identificación del Problema (Documento breve que describa):**
* Situación actual.
* Consecuencias.
* Oportunidad de mejora.


4. **Usuarios Objetivo (Definir):**
* Usuarios principales.
* Necesidades.
* Características relevantes.


5. **Investigación Preliminar (Analizar):**
* Soluciones existentes.
* Competidores.
* Diferenciadores.





### Semana 12 – Sprint 0 (Planificación)

* **Objetivo:** Construir la visión del producto y planificar el desarrollo.
* **Entregables:**
1. **Product Vision (Debe incluir):**
* Problema.
* Solución.
* Usuarios.
* Beneficios.


2. **Propuesta de Valor:** Utilizar una plantilla simple:

$$\text{Cliente} \rightarrow \text{Problema} \rightarrow \text{Solución}$$


3. **Identificación de Actores (Ejemplo):**
* Cliente
* Administrador
* Operador
* Invitado


4. **Product Backlog:** Mínimo 10 Historias de Usuario.
5. **Historias de Usuario:** Formato estándar (*Como [tipo de usuario] quiero [funcionalidad] para [beneficio]*).
* *Ejemplo:* Como cliente quiero registrarme para acceder a los servicios de la plataforma.


6. **Sprint Backlog Inicial:** Historias seleccionadas para el Sprint 1.
7. **Mockups o Wireframes (Mínimo):**
* Pantalla principal.
* Login.
* Pantalla principal del negocio.





### Semana 13 – Sprint 1 (MVP)

* **Objetivo:** Construir un producto mínimo funcional.
* **Entregables:**
1. **Configuración del Proyecto:** Angular, React o JavaScript puro. GitHub configurado.
2. **Navegación:** Debe existir navegación funcional entre vistas.
* *Ejemplo:* Inicio $\rightarrow$ Login $\rightarrow$ Dashboard


3. **Diseño Inicial (Aplicación con):** Header, Menú y Diseño consistente.
4. **Primer Flujo Funcional:**

$$\text{Registrar usuario} \downarrow \text{Visualizar usuario}$$




* **Evidencia:** Demostración en ejecución.

### Semana 14 – Sprint 2 (Gestión del Negocio)

* **Objetivo:** Implementar la lógica principal del negocio.
* **Entregables:**
1. **CRUD Principal:** Debe permitir Crear, Consultar, Actualizar y Eliminar información relevante.
2. **Validaciones (Ejemplos):** Campos obligatorios, Formatos válidos, Restricciones de negocio.
3. **Persistencia (Opciones):** LocalStorage, JSON Server o API REST.
4. **Consumo de APIs:** Utilizando `fetch()` o `HttpClient`.



### Semana 15 – Sprint 3 (Integración y Optimización)

* **Objetivo:** Completar el flujo de negocio.
* **Entregables:**
1. **Integración de Módulos:** Todos los módulos deben interactuar correctamente.
2. **Manejo de Errores:** Mostrar mensajes adecuados al usuario. *Ejemplo: "Error al guardar registro."*
3. **Optimización UX:** Mejorar la navegación, accesibilidad y diseño visual.
4. **Dashboard o Reportes (Mostrar indicadores):**
* *Ejemplo:* Total de usuarios, Total de registros, Actividad reciente.





### Semana 16 – Sprint 4 (Calidad y Despliegue)

* **Objetivo:** Preparar una versión candidata para producción.
* **Entregables:**
1. **Corrección de Defectos:** Resolución de errores reportados.
2. **Refactorización:** Mejorar organización, legibilidad y mantenibilidad del código.
3. **Validación de Requisitos:** Verificar historias completadas y flujo completo de negocio.
4. **Despliegue (Opciones):** GitHub Pages, Netlify o Vercel. *El sistema debe estar accesible mediante URL pública.*



### Semana 17 – Presentación Final

* **Objetivo:** Demostrar el producto terminado y justificar las decisiones tomadas.
* **Parte 1 – Negocio:**
* **Problema:** ¿Qué necesidad existe?
* **Solución:** ¿Cómo la aplicación resuelve el problema?
* **Innovación:** ¿Qué la diferencia de otras soluciones?
* **Usuarios Beneficiados:** ¿Quiénes obtienen valor?


* **Parte 2 – Técnica:**
* **Arquitectura:** Explicar componentes, módulos y flujo general.
* **Tecnologías Utilizadas:** Justificar la elección de Angular / React / JavaScript, APIs utilizadas y herramientas complementarias.
* **Flujo de Negocio:** Explicar el recorrido principal del usuario.
* **Retos Encontrados:** Problemas y soluciones implementadas.


* **Parte 3 – Demostración:**
* Mostrar en vivo: Registro/Login, Operaciones CRUD, Dashboard, Consumo de APIs y Flujo principal de negocio completo.
* *La demostración deberá realizarse obligatoriamente sobre la aplicación desplegada y no únicamente desde el entorno local.*

## 7. Formato de Historias de Usuario (Criterios INVEST)

Las historias de usuario deberán redactarse siguiendo estrictamente el formato:

$$\text{Como [tipo de usuario], quiero [funcionalidad], para [beneficio de negocio].}$$

Además, deben cumplir con los criterios **INVEST**:

* **I**ndependent (Independiente)
* **N**egotiable (Negociable)
* **V**aluable (Valiosa)
* **E**stimable (Estimable)
* **S**mall (Pequeña)
* **T**estable (Testeable)
